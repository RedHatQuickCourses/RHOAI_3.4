= Workload Variant Autoscaler (Inference-Aware Autoscaling) for llm-d Model Deployments (Developer Preview)

== What is it?

The Workload Variant Autoscaler (WVA) is a Kubernetes controller that provides intelligent, inference-aware automatic scaling for llm-d model deployments. Unlike traditional autoscalers that rely on generic CPU or memory metrics, the WVA derives scaling decisions from inference-specific signals—specifically KV cache utilization and request queue length—to accurately reflect the GPU-bound state of the model server.

== Why does it matter?

Traditional metrics (CPU/RAM) do not represent the true load of LLM inference, leading to over-provisioning or performance degradation. The WVA addresses this through:

- *Token-Level Demand Estimation*: It computes demand based on tokens actively being processed (KV cache) and those waiting in the queue.
- *Saturation-Based Spare Capacity Model*: It evaluates whether the current replica set has sufficient "spare capacity" to absorb traffic increases rather than waiting for full saturation.
- *Hardware-Aware Optimization*: It coordinates scaling across variants sharing a model identifier, scaling up the "cheapest" variant (e.g., A100 vs H100) first to optimize infrastructure costs.

== How to use it?

- *Enable the Controller*: You must first enable the WVA in the *DataScienceCluster custom resource*.
- *Define Performance Triggers*: Configure *kvCacheThreshold* and *queueLengthThreshold* within your *LLMInferenceService* configuration.
- *Actuation*: The WVA writes the *wva_desired_replicas* metric to Prometheus, which KEDA then reads to actuate the Horizontal Pod Autoscaler (HPA).
- *Safety Features*: The system includes a scale-down safety check that simulates load redistribution to prevent premature down-scaling, and skips variants with "pending" replicas to prevent over-provisioning during model loading.
